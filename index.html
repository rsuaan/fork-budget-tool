ℹ️ What This Tool Can and Cannot Tell You

Click Analyze above to auto-populate features from OTEL logs. If auto-detect is unavailable, gather the info from the customer using the questions below.

FeatureDetectable from Twilio data?Notes
Media Streams Auto-detected Detected from voice-media-streamer OTEL logs via LogAttributes['session_id'] (MZ SIDs). Track mode defaults to both_tracks = 2 forks — confirm with customer if single-track.
SIPREC Auto-detected Detected from voice-siprec-service OTEL logs via LogAttributes['stream_id'] (SR SIDs). Track mode defaults to both_tracks = 2 forks — confirm with customer if single-track.
Real-Time Transcription Auto-detected Detected from realtime-transcription-service OTEL logs. Track mode (inbound/outbound channels) is extracted from the log body — 2 forks if both, 1 fork if single.
AMD Auto-detected Detected from voice-amd-service OTEL logs. Note: AMD fork is temporal — released once analysis completes. Auto-detect confirms it ran but not if it overlapped with stream creation.
Fork limit kill event Not detectable The platform emits no diagnostic when a fork is killed. Only symptom: stream-stopped fires immediately with no StreamError after a 200 from the REST API.
Questions to ask the customer:
  • Are you using Media Streams? If yes — how many, and are they both_tracks or single track?
  • Are you using SIPREC? If yes — default config (both_tracks) or single track?
  • Are you using <Transcription>? If yes — which track mode?
  • Is AMD (MachineDetection=Enable) active on this call?
  • Are streams being created at the start of the call, or after AMD completes?
